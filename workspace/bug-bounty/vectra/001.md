# BUG BOUNTY REPORT: Vectra AI - support.vectra.ai

## Vulnerability: Clickjacking (CSP frame-ancestors *) on Salesforce Community Login Page

**Severity:** Medium
**Asset:** `support.vectra.ai` (covered by `*.vectra.ai` wildcard scope)
**Date:** 2026-06-21
**Header sent:** `X-Bug-Bounty: portal-username=finch`

---

## Summary

The Salesforce community at `support.vectra.ai` has a Content-Security-Policy header with `frame-ancestors *` on all pages, including the login page at `/vectra/login`. There is no `X-Frame-Options` header to provide defense-in-depth. This allows any external website to embed the Vectra support portal in an iframe, enabling clickjacking attacks against authenticated users.

Additionally, the `SelfRegister` endpoint leaks internal Salesforce infrastructure identifiers when sent a POST request without authentication.

---

## Findings

### 1. Clickjacking via CSP `frame-ancestors *`

**Affected URLs:**
- `https://support.vectra.ai/vectra/login` (login page - highest impact)
- `https://support.vectra.ai/vectra/` (home page)
- All community pages under `/vectra/`

**Evidence:**

The CSP header on the login page contains:
```
frame-ancestors *
```

No `X-Frame-Options` header is present on any response.

**Impact:**
An attacker can create a malicious page that embeds `https://support.vectra.ai/vectra/login` in a transparent iframe. A victim user who is already authenticated (or who enters credentials) can be tricked into interacting with the framed login form. Since the Salesforce community handles support cases, contacts, and downloads, clickjacking could be used to:
- Trick users into submitting support cases with attacker-controlled content
- Intercept credential entry via UI redressing
- Trick users into performing actions on their support account

**Proof of Concept:**
```html
<!DOCTYPE html>
<html>
<head><title>Clickjacking PoC - Vectra Support</title></head>
<body>
<h1>Click the button below to win a prize!</h1>
<iframe src="https://support.vectra.ai/vectra/login" 
        style="opacity:0.3; position:absolute; top:0; left:0; width:100%; height:100%; z-index:10;">
</iframe>
<button style="position:absolute; top:200px; left:150px; z-index:5;">Win Prize!</button>
</body>
</html>
```

**Remediation:**
Change CSP `frame-ancestors` from `*` to `self` (or specific trusted origins):
```
frame-ancestors 'self'
```
Additionally, add `X-Frame-Options: SAMEORIGIN` as defense-in-depth.

### 2. Information Disclosure via SelfRegister Endpoint

**Affected URL:** `https://support.vectra.ai/vectra/SelfRegister` (POST)

**Evidence:**

Sending a POST request with arbitrary form data to the SelfRegister endpoint returns a page containing `UserContext.initialize()` with internal Salesforce identifiers:

```
userName: vectra_support@00di0000000hoyreag.org.force.com
userId: 005Rg000007eWUz
networkId: 0DBRg00000009Pt
auraDomain: vectranetworks.lightning.force.com
vfDomainPattern: vectranetworks--(?:[^.]+).vf.force.com
siteUrlPrefix: /vectravforcesite
```

**Reproduction:**
```bash
curl -X POST -H "X-Bug-Bounty: portal-username=finch" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "firstName=test&lastName=test&email=test@test.com" \
  "https://support.vectra.ai/vectra/SelfRegister"
```

**Impact:**
- Exposes internal Salesforce org ID (`00di0000000hoyreag`), user ID, network ID
- Reveals internal domain patterns: `vectranetworks.lightning.force.com`, `vectranetworks--*.vf.force.com`
- Reveals the default community user email pattern: `vectra_support@00di0000000hoyreag.org.force.com`
- These identifiers can be used for targeted attacks against the Salesforce organization

---

### 3. Public Salesforce API Version Listing

**Affected URL:** `https://support.vectra.ai/services/data`

**Evidence:**
```bash
curl -H "X-Bug-Bounty: portal-username=finch" "https://support.vectra.ai/services/data"
```

Returns a JSON array listing all 37 available Salesforce API versions (v31.0 through v67.0, Summer '26). While data queries require authentication (`Authorization required`), the version listing is accessible without credentials, revealing the Salesforce platform version.

## Additional Observations (Lower Priority)

- **Weglot API Key Exposure:** The Weglot translation API key `wg_4ee8d93119cf98e624af53693260d0858` is exposed in client-side JavaScript on `www.vectra.ai`. The `project-settings` endpoint is accessible and returns project configuration, excluded paths, and language settings. This is partially by-design for Weglot's client-side integration, but exposes internal site structure.
- **S3 Bucket Listing:** The `vectra` S3 bucket (`vectra.s3.amazonaws.com`) is publicly listable, though it only contains old images from 2010 (appears to be a different entity's bucket, not Vectra AI's).
- **Internal Infrastructure in CSP:** The CSP header on `support.vectra.ai` reveals `jira.vectra.io`, `vectranetworks.my.salesforce-scrt.com`, `vectranetworks--krow.vf.force.com`, and `storage.googleapis.com` as trusted domains.

## Scope Confirmation

- `support.vectra.ai` matches the `*.vectra.ai` wildcard scope entry
- `eligible_for_submission: true`
- `max_severity: critical`
- `partners.vectra.ai` is explicitly OUT OF SCOPE (but `support.vectra.ai` is not)

---

## Recommendations

1. **Immediate:** Change `frame-ancestors *` to `frame-ancestors 'self'` in the Salesforce community CSP configuration
2. **Immediate:** Add `X-Frame-Options: SAMEORIGIN` header
3. **Short-term:** Restrict the `SelfRegister` endpoint to not return `UserContext` data for unauthenticated POST requests
4. **Short-term:** Restrict `/services/data` endpoint to authenticated users only
5. **Consider:** Move the Weglot API key to a server-side proxy rather than exposing it in client-side JS
